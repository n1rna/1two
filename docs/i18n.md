# Internationalization (i18n)

How translation works in the kim app. If you're adding new user-visible
text, read this.

## Architecture

- **Library**: `react-i18next` + `i18next` + `i18next-browser-languagedetector`
- **Mode**: client-only. All translations are statically imported and
  bundled by webpack. No server-side i18n, no filesystem access, works on
  Cloudflare Workers.
- **No URL-based routing**: locale is NOT part of the URL. No `/fr/today`.
  Locale is detected from the browser and stored in `localStorage`.
- **Fallback**: English (`en`) is the fallback for any missing key.

## How it's wired

`I18nProvider` (in `apps/kim/src/components/layout/i18n-provider.tsx`)
wraps the entire app at the root layout level. It provides the i18next
instance to all components and syncs `document.documentElement.lang`
whenever the language changes.

The provider sits between `ThemeProvider` and `QueryProvider`:

```
ThemeProvider > I18nProvider > QueryProvider > TooltipProvider > {children}
```

Locale detection order:
1. `localStorage` key `kim-locale` (set when user picks a language)
2. `navigator.language` / `navigator.languages`
3. Falls back to `en`

## How to use in components

Every `"use client"` component uses the `useTranslation` hook:

```tsx
import { useTranslation } from "react-i18next";

export function MyComponent() {
  const { t } = useTranslation("health"); // loads the health namespace

  return (
    <div>
      <h1>{t("title")}</h1>
      <p>{t("subtitle")}</p>
      <span>{t("weight_display", { value: 75.5 })}</span>
    </div>
  );
}
```

For multiple namespaces:
```tsx
const { t } = useTranslation(["health", "common"]);
// t("health:title") or t("common:back")
```

## Token naming rules

- Keys are **flat snake_case** within each namespace. The namespace
  provides the grouping â€” no nested objects.
- Descriptive names: `empty_title`, `ask_kim_prompt_body`,
  `activity_sedentary`, not `s1`, `msg3`.
- Interpolation uses `{{variable}}`: `"weight_display": "{{value}} kg"`.
- Labels for enum-like maps use a prefix: `activity_sedentary`,
  `activity_light`, `gender_male`, etc.

## Adding a new translatable string

1. **Pick the namespace** that matches the feature/page the string appears in.
   If it's shared across multiple pages, put it in `common`.

2. **Add the key to `en/<namespace>.json`**:
   ```json
   {
     "my_new_label": "Some visible text"
   }
   ```

3. **Add the same key to every other locale** (`fr`, `nl`, `de`, `es`,
   `it`, `pl`). Initially copy the English value â€” translators will fill
   in the real translation. If you skip this, the completeness test fails.

4. **Use it in the component**:
   ```tsx
   const { t } = useTranslation("my-namespace");
   return <span>{t("my_new_label")}</span>;
   ```

5. **Run the completeness test**:
   ```bash
   cd apps/kim && bun run test:i18n
   ```

## Adding a new namespace

1. Create `apps/kim/src/i18n/locales/en/<namespace>.json`.
2. Create the same file in all 6 other locale directories.
3. Add static imports in `apps/kim/src/i18n/config.ts` for every locale.
4. Add the namespace name to the `NAMESPACES` array in `config.ts`.
5. Add the namespace to the `ns()` helper function parameters and all
   `resources` entries.

## Adding a new language

1. Create `apps/kim/src/i18n/locales/<code>/` with all 16 JSON files
   (copy from `en/` as a starting point).
2. Add all static imports in `config.ts`.
3. Add the locale code to `SUPPORTED_LOCALES` and `LOCALE_LABELS`.
4. Add the locale's resource entry in the `resources` object.
5. The completeness test automatically picks it up.

## Completeness test

```bash
cd apps/kim && bun run test:i18n
```

This runs a vitest suite that:
- For every namespace, for every non-English locale, verifies that every
  key in `en/<ns>.json` exists in `<locale>/<ns>.json`.
- Also verifies there are no extra keys in any locale that don't exist
  in English (catches stale keys after renames).
- Fails CI if any key is missing or extraneous.

Currently: 192 tests (16 namespaces x 6 non-English locales x 2 checks).

## Locale persistence and the language picker

The user picks their language in Settings (`/settings`). The picker calls
`i18n.changeLanguage(locale)`, which:
- Switches all `t()` calls immediately (React re-renders).
- Persists the choice to `localStorage` key `kim-locale`.
- Updates `document.documentElement.lang`.

On next visit, `i18next-browser-languagedetector` reads `kim-locale`
from localStorage before falling back to the browser's language.

## What stays in English

- **Metadata** (page titles, OpenGraph, Twitter cards): these are
  server-side `export const metadata` and cannot use `useTranslation`.
  They stay in English since we don't do URL-based locale routing.
- **API-returned content**: routine names, meal plan titles, kim chat
  messages â€” these are user-generated or AI-generated in the user's
  language and aren't translated through i18n.
- **Technical labels**: tool names like `calendar.move`, CSS classes,
  route paths.
